# CA Research — пожелания и требования владелицы проекта

_Собрано 9 июля 2026. Живой документ: дополняется по ходу работы и наладки._

## 1. Контроль процесса исследования (human-in-the-loop)

Раньше весь процесс шёл вручную через GPT — каждый шаг проверялся и правился.
В приложении модули бегут автоматически, и ошибку в середине не поймать.

Требуется:
- **Стоп-точки после модулей** — режим «с проверкой»: модуль закончился → показать
  результат → «Дальше» или «Переделать с комментарием».
- **Редактирование результата модуля** прямо в приложении (например, поправить
  формулировку боли у персоны) — следующие модули опираются на исправленную версию.
- **Re-run с замечанием** — перезапуск модуля с конкретным указанием, что исправить.
- Особо критичные точки контроля: M3 (VoC) и M4 (персоны, JTBD, интервью).
- **Согласование пиковых точек исследования.** В ключевых развилках приложение
  не решает само, а задаёт вопрос заказчику. Пример: выявили ниши → нейросеть
  рекомендует перспективные → но заказчик может не согласиться и выбрать другую
  нишу для дальнейшего исследования. Выбор ниши (и подобные развилки) — всегда
  за человеком; рекомендация ИИ — только подсказка. Аналогично уже работает
  выбор ценового слоя после M1.

## 2. Фиктивные интервью (M3/M4)

- «Интервьюирование» персон — выдуманное, но на основе проведённого анализа.
- Продумать: сколько лиц на сегмент, у каждого свои боли, свои JTBD.
- Логика — в исходной методологии (файл с промптами, 22 блока). Ждём файл
  от владелицы для разбора.

## 3. Исследование — фундамент для контента

Результаты исследования используются дальше:
- посадочные страницы под каждую боль/услугу (одна страница может закрывать несколько болей);
- контент-машина берёт данные из исследования;
- SEO-продвижение основного сайта — тоже на основе исследования.

Значит: результаты модулей хранить не только как текст отчёта, но и как
**структурированные данные** (персоны, боли, JTBD, сегменты, офферы — JSON),
чтобы будущие агенты (лендинги, контент, SEO) могли брать их напрямую.

## 4. Языковая система: язык специалиста ↔ язык исследования

Сервисом будут пользоваться специалисты из разных стран. У каждого — своя пара языков:

- **Язык специалиста** (рабочий язык того, кто делает и контролирует исследование) —
  настраивается свободно: русский, английский, турецкий, любой. На нём: интерфейс,
  чтение результатов, стоп-точки, принятие решений.
  Пример владелицы: живёт в Турции, работает по-русски, турецким не владеет.
- **Язык исследования** (язык страны рынка) — на нём само исследование: подлинные
  формулировки аудитории (боли, желания, мысли), ключевые фразы для семантики.
- Результаты специалист должен **сразу читать на своём языке** (перевод),
  но в **контент-машину** данные идут на языке оригинала (не переведённые
  туда-обратно) — иначе теряются подлинные формулировки.
- Нужен удобный механизм показа: оригинал + перевод рядом (переключатель или
  параллельно) — продумать при реализации. Ранее пробовали — было неудобно;
  владелица напомнит детали, когда начнём.

**Текущая проблема:** в приложении хаос из трёх языков — часть интерфейса
на английском, часть на русском, исследование пытались делать на турецком.
Языковую систему нужно привести к чистой модели «язык специалиста + язык
исследования», без жёстко зашитого набора.

## 5. Отчёты — для руководителя, презентабельный PDF

- Сырые файлы исследования большие и для руководителя нечитабельные.
- Отчёт должен быть: презентабельный, читабельный, PDF.
- Главное в подаче — **выводы**: что проделали, какие данные получили,
  что с ними можно сделать.
- Отчёт по M1 «никак не получался» — переделать.
- **Процесс:** дизайн отчёта согласовывается с владелицей по каждой части
  отдельно, до закрепления в коде. Возможна разработка дизайн-системы.

## 6. Конструктор воронок продаж (идея, дальний горизонт — НЕ в текущее ТЗ)

Записано 12 июля 2026. После аналитического слоя (см. ТЗ-АНАЛИТИЧЕСКИЙ-СЛОЙ.md) рассмотреть
модуль «Конструктор воронок» (условно M11):
- вход: персоны и боли (M4), офферы (M5), контент-система (M6);
- выход: 2–3 варианта воронки под сегмент (трафик → прогрев → конверсия)
  с точками замера — для тестирования контент-машиной; сравнение воронок — в трекере M10.
- Методологическая база — материалы Никиты Корытина (у владелицы).
  Важно: если сервис станет публичным продуктом, методологию переработать
  в собственную (чужой курс нельзя зашивать внутрь как есть).

## Решения обсуждения проекта 12 июля 2026 (архитектура исследования)

- **Вариант А**: CA Research — рабочий инструмент владелицы (не SaaS). «Сервис
  для специалистов из разных стран» — возможность на потом, не требование v1.
  Требование: анализ любого рынка + понятный владелице результат (языковая пара).
- **M1 делится на два модуля**:
  - **M1.1 «Ёмкость рынка»** (блоки 01–03: гео-статистика, TAM/SAM/SOM, доли) —
    первый в цепочке, но скрыт/заблокирован до реальных источников статистики;
  - **M1.2 «Разведка ниш»** (блоки 04, 04_1, 04_2) — активный первый шаг:
    карта ниш со скорингом → стоп-точка, нишу выбирает человек (обычно заказчик
    «тыкает пальцем» → экран презентабельный, карточки как у ценовых слоёв).
    Выбор ниши включает «ФИЛЬТР НИШ» из методологии во всех следующих модулях.
- **Чек-лист услуг из сайта**: автозаполнение брифа дополнительно извлекает
  список услуг/направлений (тот же GPT-вызов) → чекбоксы в брифе. Клиент выбрал
  услугу (кейс «ботокс») → разведка узкая; не выбрал → M1.2 скорит шорт-лист
  + 1–2 идеи от ИИ с пометкой «предложено ИИ».
- **Воронки — после M5, перед M6**: порядок выполнения
  M1.2 → (M1.1 спит) → M2 → M3 → M4 → M5 → Воронки → M6 → M7 → M8–M10.
  Контент-система (M6) наполняет этапы выбранной воронки, SEO (M7) ведёт трафик.
  Вариант воронки выбирает человек (стоп-точка).
- **Поиск — Tavily** (не Perplexity): возвращает сырые выдержки с URL → полный
  контроль заземления; Perplexity — уже синтезированный ответ чужой модели.
  Бесплатного уровня (~1000 запросов/мес) на старте хватит.
- **Формат стоп-точек** под стиль владелицы: короткая выжимка (5–7 строк) +
  красные флаги по низкому Confidence + выборочное раскрытие деталей.
  Не редактирование текстов, а «адекватно / переделай вот это».
- **Улучшения методики** (аудит 12 июля, вшивать в промпты по ходу):
  правило свежести (источники >12 мес — пометка и пониженный вес); поправка
  на перекос площадок; шкала интенсивности боли рядом с частотностью;
  «что пробовали до» (alternatives); флаг противоречий (говорят ≠ делают).
  Спиральная динамика — уже в методике (блок 09, SD-уровень) как опция для
  тона коммуникации; осью сегментации не делать.
- **Дизайн**: текущий вид не устраивает. Этап дизайна: 2–3 варианта
  дизайн-направления (макеты в браузер) → выбор владелицы → дизайн-система
  (интерфейс + дашборды + отчёты) → перенос по экранам, каждый после «да».
  Направления можно делать параллельно первому спринту; редизайн экранов —
  после инфраструктуры.
- **Первый спринт**: защита доступа → Tavily-эндпоинт → M1.2 Разведка ниш
  с чек-листом услуг → БД Supabase → дальше по ТЗ (M10, стоп-точки, M8, M9, воронки).
