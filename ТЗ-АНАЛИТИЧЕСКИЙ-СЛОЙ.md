## 1. Зачем это нужно (контекст)

CA Research сейчас умеет делать **разовый снимок** аудитории (модули M1–M7: рынок, конкуренты, VoC, психология, офферы, контент-система, SEO). Этого достаточно для старта проекта, но недостаточно для его ведения.

Появился реальный потребитель данных — **контент-машина** (первый кейс: продвижение сервиса LeadYup на рынке РФ, цель — регистрации). Машина работает недельными циклами и требует данных, которые **обновляются во времени**:

- что нового в нише за неделю (тренды, инфоповоды);
- какой контент конкурентов залетает и почему;
- как отработали собственные публикации (просмотры → переходы → регистрации).

Итоговая петля: **исследование → контент → метрики → обратно в исследование.** Замкнутая петля — главное конкурентное отличие CA Research; ни у одного из известных аналогов её нет.

## 2. Блокирующие инфраструктурные предпосылки

Без этих трёх пунктов модули M8–M10 будут декорацией. Делать **до** модулей.

### 2.1. Реальный веб-поиск (приоритет №1)

Сейчас промпты M2/M3 требуют «only verifiable open sources with real URLs», но GPT-4.1 через `api/proxy.js` не имеет доступа в интернет — «источники» и «цитаты» генерируются из памяти модели, т.е. частично галлюцинации. Для трендов (M8) это фатально: модель выдаст тренды годичной давности.

Требуется:
- Подключить поисковый API (Tavily — уже в планах проекта; альтернатива Perplexity API) через отдельный serverless-эндпоинт рядом с `api/proxy.js`.
- Пайплайн: поиск → передача реальных выдержек с URL в промпт модуля → модель работает только с переданным материалом, запрет «дополнять из памяти».
- Внедрить сначала для M8/M9 (обязательно), затем ретрофит в M2/M3 (желательно).

### 2.2. База данных вместо localStorage

Сейчас всё хранится в `localStorage` браузера: теряется при чистке, недоступно с другого устройства и недоступно внешним агентам. Аналитический слой — это накопление истории неделями, в localStorage он невозможен.

Требуется:
- Подключить БД (Supabase или Vercel Postgres — бесплатных тарифов достаточно).
- Хранить: результаты прогонов модулей (структурированный JSON, не только текст отчёта), снапшоты M8/M9 по датам, журнал публикаций и метрик M10.
- Существующий экспорт `agent_package.json` — правильный зачаток: его структура становится основой схемы БД.
- Доступ для внешних агентов: простейший REST-эндпоинт или прямой доступ к Supabase по ключу (read-only токен для чтения, отдельный ключ для записи метрик).

### 2.3. Периодические прогоны

Архитектура сейчас — «нажал кнопку → один прогон». M8 и M10 — процессы по расписанию.

Требуется:
- Vercel Cron (или ручная кнопка «Обновить неделю» на первом этапе — допустимо).
- У каждого прогона M8/M9 — дата; модуль обязан видеть предыдущий снапшот и считать **дельту**.

## 3. Модуль M8 — Тренд-монитор

**Вопрос модуля:** что изменилось в нише за период?

**Входы:** бриф проекта (ниша, рынок, язык исследования), список ключевых тем/запросов (можно генерировать из M3/M7), предыдущий снапшот M8.

**Что делает:** еженедельный скан через реальный поиск (2.1): инфоповоды ниши, растущие темы и запросы, новые игроки/продукты, изменения у конкурентов из M2, обсуждаемые боли аудитории (свежие треды форумов, соцсетей).

**Выходы (структурированные, в БД):**
- `trends[]`: тема, динамика (новая/растёт/затухает), источники (URL), рекомендация для контента;
- `newsworthy[]`: инфоповоды с датой актуальности («сгорает до…»);
- `delta`: сравнение с прошлым снапшотом — что появилось, что умерло.

**Периодичность:** еженедельно.

## 4. Модуль M9 — Контент-радар

**Вопрос модуля:** какой контент в нише реально работает?

Отличие от M2: тот смотрит на конкурентов как на **бизнесы** (цены, офферы, позиционирование), M9 — как на **медиа** (что и как они публикуют, что залетает).

**Входы:** список каналов конкурентов и смежных ниш (соцсети, YouTube, Telegram) — вводится вручную + пополняется из поиска; предыдущий снапшот.

**Что делает:** сбор публично видимых показателей (просмотры, лайки, комментарии — то, что видно без API площадок), разбор топ-контента на составляющие: тема, формат, хук первых секунд, длительность, CTA, персонаж/лицо.

**Выходы (структурированные):**
- `channels[]`: канал, площадка, подписчики, динамика;
- `top_content[]`: единица контента, показатели, разбор (тема/формат/хук/CTA), URL;
- `patterns[]`: обобщения — «в нише работают форматы X, хуки типа Y, длительность Z»;
- бенчмарки: медианные показатели ниши (для сравнения своих результатов в M10).

**Периодичность:** раз в 2–4 недели.

**Честное ограничение:** без официальных API площадок сбор частично ручной/полуавтоматический (агент получает ссылки и скриншоты статистики). Проектировать так, чтобы источник данных мог смениться с ручного на API без смены схемы.

## 5. Модуль M10 — Трекер публикаций + дашборд

**Вопрос модуля:** что мы опубликовали и что это дало?

### 5.1. Схема данных (контракт с контент-машиной — согласован, не менять без обсуждения)

Таблица `publications`:

| Поле | Тип | Пример |
|---|---|---|
| id | uuid | — |
| date | date | 2026-07-20 |
| platform | enum | youtube_shorts / vk_clips / telegram / instagram_reels / instagram_carousel / dzen / rutube / max / pinterest / tiktok |
| format | enum | short_video / long_video / carousel / text_post |
| character | enum | rabbit / owner / none |
| topic | text | «3 ошибки в попапах» |
| hypothesis_id | text | H-03 (гипотеза из M5 или теста форматов) |
| hook_type | text | вопрос / шок-факт / история… |
| url | text | ссылка на публикацию |
| utm | text | utm_campaign=… |

Таблица `metrics_snapshots` (одна публикация — много снапшотов по датам):

| Поле | Тип |
|---|---|
| publication_id | uuid |
| snapshot_date | date |
| views, likes, comments, shares | int |
| watch_pct (досмотры, %) | float |
| subs_gained | int |
| clicks (переходы по UTM) | int |
| registrations (из аналитики сайта) | int |

Ввод метрик на первом этапе — ручной (форма или импорт CSV): владелица раз в неделю переносит цифры из статистики площадок. Проектировать под будущий импорт из API.

### 5.2. Дашборд

Chart.js уже в проекте — использовать его. Виджеты:
- динамика по неделям: публикации / просмотры / переходы / регистрации;
- сравнение форматов (short vs carousel vs text) и персонажей (заяц vs собственник vs без) по медианным показателям — это главный инструмент теста гипотез;
- сравнение площадок: где живёт аудитория, откуда регистрации;
- воронка: просмотры → переходы → регистрации, по неделям;
- сравнение своих показателей с бенчмарками ниши из M9.

## 6. Контракт с контент-машиной

- Контент-машина **читает**: результаты M2–M7 (сегменты, боли, VoC, офферы), снапшоты M8/M9. Формат — структурированный JSON (расширение текущего `agent_package.json`), на **языке исследования** (оригинальные формулировки аудитории, без перевода — см. ПОЖЕЛАНИЯ.md п. 4).
- Контент-машина **пишет**: строки в `publications` (при публикации) и `metrics_snapshots` (еженедельно).
- Пока БД не готова, временный интерфейс — файлы/таблица на стороне контент-машины по этой же схеме; при готовности M10 данные импортируются. **Схему из 5.1 менять только по согласованию** — на неё уже завязан второй проект.

## 7. Доработки существующих модулей M1–M7

Консолидировано из ПОЖЕЛАНИЯ.md — детали и мотивировка там.

### 7.1. Human-in-the-loop (контроль процесса)

Сейчас модули бегут автоматически, ошибку в середине не поймать. Требуется:
- **Стоп-точки** — режим «с проверкой»: модуль закончился → показать результат → «Дальше» или «Переделать с комментарием».
- **Редактирование результата модуля** прямо в приложении; следующие модули опираются на исправленную версию.
- **Re-run с замечанием** — перезапуск модуля с конкретным указанием, что исправить.
- **Развилки — за человеком:** в ключевых точках (выбор ниши, выбор ценового слоя) приложение задаёт вопрос, а не решает само; рекомендация ИИ — только подсказка.
- Критичные точки контроля: M3 (VoC) и M4 (персоны, JTBD).

### 7.2. Языковая система

Привести к чистой модели двух независимых языков (вместо текущего хаоса из трёх):

- **Язык специалиста** — родной язык того, кто ведёт исследование (у владелицы — русский). На нём: интерфейс, стоп-точки, принятие решений. Специалист может вести исследования разных рынков (США, Турция и др.), не владея их языками.
- **Язык исследования** — язык рынка. На нём модули **создают и хранят** все результаты: подлинные формулировки аудитории, цитаты VoC, семантику. Это всегда оригинал.

Механика перевода — **кнопка «Перевести» по запросу** у результата модуля: нажала → прочитала на своём языке → продолжила. Ключевые правила:
- перевод — только слой для чтения; он **не сохраняется** в рабочие данные и не подменяет оригинал;
- в следующие модули, экспорт и контент-машину уходит **только оригинал** (перевод туда-обратно убивает подлинные формулировки);
- редактирование результата (7.1) происходит в оригинале; если специалист правит через перевод — механизм согласовать с владелицей отдельно.

Владелица долго прорабатывала эту модель в переписке — при реализации сверить детали с ней до закрепления в коде.

### 7.3. Фиктивные интервью (M3/M4)

«Интервьюирование» выдуманных, но основанных на данных персон: сколько лиц на сегмент, у каждого свои боли и JTBD. Логика — в методологии владелицы (файл с промптами, 22 блока); файл запросить у неё до реализации.

### 7.4. Отчёты — презентабельный PDF

Сырые отчёты для руководителя нечитабельны. Требуется PDF с акцентом на **выводы** (что проделали → что получили → что с этим делать). Дизайн каждой части согласовывается с владелицей до закрепления в коде. Отчёт M1 ранее «не получался» — переделать при возврате M1.

### 7.5. Ретрофит аналитической инфраструктуры

- Реальный поиск (2.1) подключить к M2 (конкуренты) и M3 (VoC-цитаты) — сейчас там частично галлюцинации.
- Результаты всех модулей сохранять в БД (2.2) как структурированный JSON, а не только текст отчёта.

## 9. Порядок работ и приоритеты

1. **2.1 Веб-поиск** — разблокирует всё, плюс лечит галлюцинации существующих M2/M3.
2. **2.2 База данных** — начать со схемы 5.1 (publications/metrics) и сохранения результатов модулей.
3. **M10 трекер** (ввод + хранение, без красоты) → **M10 дашборд** (+ данные Метрики/GA4).
4. **7.1 Human-in-the-loop** — стоп-точки, редактирование, re-run: поднимает качество всех исследований, идёт параллельно слою аналитики.
5. **M8 Тренд-монитор.**
6. **M9 Контент-радар.**
7. Ретрофит поиска в M2/M3 + фиктивные интервью 7.3 (после получения файла методологии).
8. **7.2 Языковая система.**
9. **7.4 PDF-отчёты** — последними: до них дизайн-система и согласование с владелицей.

Контент-машина стартует, не дожидаясь этих доработок (первые недели — на временной таблице), поэтому дедлайн мягкий, но порядок важен: к моменту, когда у машины накопится 4–6 недель метрик, M10 должен уметь их принять.

## 10. Анти-цели (что сознательно НЕ делаем)

- Не подключаем официальные API соцсетей на первом этапе (дорого по времени, у части площадок недоступно) — ручной ввод приемлем.
- Не делаем мультитенантность/аккаунты — сервис пока для одного пользователя.
- Не трогаем M1 (отложен решением от 9 июля 2026).
- Не строим «риалтайм» — недельный шаг данных достаточен.

## 11. Критерии приёмки

- [ ] M8 выдаёт еженедельный снапшот с дельтой и реальными URL источников (проверка: URL открываются, даты свежие).
- [ ] M9 выдаёт разбор топ-контента ≥5 каналов ниши с бенчмарками.
- [ ] M10: публикация вводится за ≤1 минуту, метрики — импортом CSV или формой; дашборд показывает воронку и сравнение форматов/персонажей/площадок.
- [ ] Внешний агент может прочитать данные исследования и записать метрики без ручного копирования (эндпоинт или доступ к БД).
- [ ] Данные переживают чистку браузера и открываются с другого устройства.
- [ ] Любой модуль можно остановить, отредактировать результат и перезапустить с замечанием; развилки (ниша, ценовой слой) решает человек.
- [ ] Результаты создаются и хранятся на языке исследования; кнопка «Перевести» показывает их на языке специалиста; в экспорт и следующие модули уходит только оригинал.
- [ ] Отчёт по исследованию — читабельный PDF с выводами, дизайн утверждён владелицей.
